# Spec: Config Integrity Protection

**Date:** 2026-03-31
**Finding:** Trial Run Finding 1 — config writable from inside container
**Priority:** CRITICAL

---

## Problem

The agent's config (`~/.openclaw/openclaw.json`) is writable from inside the container. If an attacker gains arbitrary code execution (bypassing Layer 3 tool policy), they can modify security settings and OpenClaw hot-reloads the changes.

## Why Full Prevention Is Hard

- OpenClaw rewrites its own config on startup (adds meta, compaction, auth tokens)
- OpenClaw watches the config file for hot-reload (feature, not bug)
- The persistent volume needs read-write for workspace, sessions, pairing
- Making the config read-only would break OpenClaw's startup sequence
- Changing file ownership to root doesn't work — Podman rootless maps everything to the container user

## Chosen Approach: Detection + Prevention Layers

### Layer 1: Config Integrity Hash (Detection)

When `tool-control.sh --apply` generates and installs a config, it computes a SHA-256 hash of the security-critical fields (tools.deny, tools.exec, tools.elevated, tools.profile) and stores it in a file on the HOST side (not inside the container).

`verify.sh` reads the running config, computes the same hash, and compares. If they differ, FAIL — the config was tampered with.

**Files:**
- Hash stored at: `.vault-config-hash` (gitignored, host-side)
- Checked by: `verify.sh` check #24 (new)
- Generated by: `tool-control.sh --apply`

### Layer 2: Entrypoint Config Lock (Prevention)

After the entrypoint installs/preserves the config and before starting OpenClaw, change the config file permissions to `444` (read-only for everyone). OpenClaw's startup will fail to rewrite it — but it reads it fine.

If OpenClaw requires write access to the config: instead of locking the whole file, lock a **shadow copy** and verify the live config against it on every verify run.

**Testing needed:** Does OpenClaw start correctly with a read-only config file? If yes, this is the strongest protection. If no, fall back to detection-only.

### Layer 3: ask: "always" (Existing Prevention)

In Split Shell, every exec command requires Telegram approval. A user who sees `node -e "...modify openclaw.json..."` would deny it. This is the primary prevention layer.

## Implementation

### Step 1: Test if OpenClaw tolerates read-only config

```bash
podman exec opencli-container sh -c "chmod 444 /home/vault/.openclaw/openclaw.json"
# Restart and check if OpenClaw starts
```

If yes → implement Layer 2 (entrypoint lock).
If no → implement Layer 1 (hash detection) only.

### Step 2: Implement whichever works

Based on Step 1 results, implement the appropriate layer.

### Step 3: Add verify.sh check #24

Config integrity check — compare running config security fields against stored hash.

## Verification

- Attempt to modify config from inside container → should fail or be detected
- `make verify` catches the tampering
- `tool-control.sh --apply` correctly stores the hash
- Normal shell switching still works
